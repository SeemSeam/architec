# Architec 架构师能力演进方案

本文档定义 `architec` 从“架构分析器”演进到“架构师审查代理”的目标能力、系统设计和落地路线。目标不是让 LLM 只会生成总结，而是让它像有经验的架构师一样，基于程序化证据持续把控项目架构发展、模块边界和工程风格。

## 1. 目标定位

`architec` 的长期目标应当是：

- 能恢复项目当前架构，而不是只统计代码指标。
- 能判断架构是否支撑未来演进，而不是只指出热点文件。
- 能持续把控架构发展方向，而不是只做一次性诊断。
- 能审查项目风格和结构一致性，而不是只看复杂度。
- 能给出分阶段治理建议，而不是只输出问题清单。

换句话说，`architec` 应当具备“程序化审查 + 架构语义建模 + LLM 架构推理”的组合能力。

## 2. 真正架构师应具备的能力

真实架构师在项目中承担的职责，不止是发现坏味道，还包括对系统演进方向负责。

### 2.1 架构恢复能力

能够从现有仓库中识别：

- 系统入口点
- 编排层、领域层、基础设施层、适配器层
- 模块职责和所有权
- 公共接口与内部实现边界
- 运行时主链路、配置链路和外部依赖链路

### 2.2 架构判断能力

能够回答这些关键问题：

- 当前结构是否支持后续功能扩展
- 目录和模块划分是否表达了清晰的能力域
- 是否存在跨层依赖、逆向依赖或万能模块
- 变更是否会在多个模块间无序扩散
- 测试结构是否与运行时结构一致

### 2.3 架构发展把控能力

真正的架构师需要持续约束项目演进，而不是事后点评。

应具备以下能力：

- 定义目标架构形态和依赖规则
- 审查新增模块是否落在正确位置
- 审查新功能是否破坏既有边界
- 控制“临时实现”演化为长期结构债务
- 对关键模块建立演进红线和例外机制

### 2.4 风格治理能力

这里的“风格”不是代码格式，而是项目级架构风格。

应覆盖：

- 命名风格是否一致
- 模块组织方式是否一致
- 新文件落点是否可预测
- 相似职责是否采用相似结构表达
- 配置、报告、适配器、编排代码是否各归其位

### 2.5 决策与权衡能力

真实架构师不会只说“应该重构”，而会判断：

- 哪些问题必须立即处理
- 哪些问题可以延期
- 重构收益是否大于扰动成本
- 哪些结构应保持稳定，不宜轻易抽象
- 推荐采用保守方案、激进方案还是分阶段方案

## 3. 当前 `architec` 的能力边界

基于当前实现，`architec` 已具备这些基础能力：

- 基于 Hippo 快照读取结构、签名和静态指标
- 聚合 findings，生成 hotspot、component risk 和评分
- 根据热点和目标做有限的功能落点建议
- 使用 LLM 生成执行摘要和优先级建议

但它仍然有明显缺口：

- 更擅长“指标聚合”，不擅长“架构恢复”
- 更擅长“热点排名”，不擅长“架构判断”
- 更擅长“事后总结”，不擅长“演进把控”
- 更擅长“文件级问题”，不擅长“仓库级结构问题”
- 更擅长“局部修复建议”，不擅长“架构权衡和路线规划”

因此它当前更像“架构质量分析器”，还不是完整意义上的“架构师代理”。

## 4. 核心设计原则

演进后的 `architec` 应遵循以下原则：

1. 先建模，后评价  
没有显式架构模型，LLM 无法稳定做架构判断。

2. 先程序化取证，后语言化推理  
LLM 不应凭印象评论，而应消费明确证据。

3. 先检查结构规则，后输出摘要  
架构审查的主体应是规则和模型，不是总结文本。

4. 评价项目演进能力，而不是只评价当前代码整洁度  
真正需要回答的是“未来还能不能稳地改下去”。

5. 保留例外机制  
架构规则要支持豁免、理由和到期时间，不能只有硬拦截。

## 5. 目标能力模型

建议将 `architec` 的架构师能力拆为六个子能力。

### 5.1 仓库拓扑审查

检查仓库是否呈现清晰的结构化组织：

- 顶层目录是否过平
- 文件是否堆在单一目录中
- 目录是否表达能力域或分层
- 是否存在孤儿模块、杂项目录、职责混装目录
- 新增文件是否有稳定落点

### 5.2 架构模型恢复

从仓库中恢复：

- 层级模型
- 组件模型
- 组件间依赖
- 入口点和主流程
- 测试与生产代码的映射关系
- 配置来源和优先级链路

输出应为结构化模型，而不是仅存在于 prompt 中。

### 5.3 架构契约审查

对恢复出的模型执行规则检查：

- 层次是否单向依赖
- 编排层是否直连底层实现
- 适配器层是否侵入领域逻辑
- 报告/展示代码是否反向污染核心逻辑
- 测试是否依赖不稳定内部模块
- 新增模块是否绕开公共接口

### 5.4 架构发展与风格治理

这部分直接对应“架构师把控项目发展和风格”的职责。

应包含：

- 目标结构模板，例如 `cli / orchestrator / analysis / scoring / backend / reporting / adapters`
- 组件命名和目录命名约束
- 新能力落点推荐
- 不鼓励出现的结构模式，例如“大而全模块”“横向工具堆积”“跨层捷径”
- 架构漂移检测，识别项目是否逐步偏离目标结构

### 5.5 风险推演与权衡

架构审查结果不应只停留在“有问题”，而应回答：

- 问题如果不处理，会在哪些需求上先爆出成本
- 处理问题需要改哪些边界
- 哪些问题适合局部重构，哪些需要结构调整
- 哪些模块是核心稳定资产，不应轻易打散

### 5.6 分阶段治理建议

输出不应是简单 TODO，而应是阶段性架构治理计划：

- `P0` 需要立即处理的阻断性结构问题
- `P1` 需要近期收敛的边界与模块问题
- `P2` 可以在后续迭代中治理的结构债务
- 对每项建议附带理由、影响面、预期收益和验证方式

## 6. 建议的系统架构

建议将 `architec` 演进为四层系统。

### 6.1 证据层

负责采集程序化证据：

- 目录拓扑
- 文件分布和模块聚类
- import 依赖图和函数依赖图
- 入口点和运行链路
- git 变更耦合
- 测试覆盖映射
- 配置流和外部依赖流
- 现有 findings 和复杂度信号

### 6.2 架构模型层

负责把证据转化为统一的架构模型：

- `layers`
- `components`
- `public_interfaces`
- `internal_modules`
- `dependency_edges`
- `runtime_paths`
- `test_mappings`
- `config_paths`
- `style_profile`

该层是整个系统的核心。没有这一层，LLM 无法稳定像架构师一样推理。

### 6.3 审查引擎层

负责用规则审查模型，输出结构性结论：

- 目录拓扑评分
- 分层清晰度评分
- 边界违规列表
- 模块归属稳定性评分
- 架构漂移评分
- 风格一致性评分
- 变更扩散风险评分

### 6.4 LLM 推理层

LLM 只消费已经结构化的模型和审查结果，负责：

- 形成架构结论
- 解释问题成因
- 做权衡分析
- 给出分阶段治理建议
- 标注结论置信度和不确定项

这时的 LLM 才真正接近“资深架构师审查”。

## 7. 关键数据产物建议

为支持上述能力，建议新增以下中间产物：

- `.architec/repo-topology.json`
- `.architec/architecture-model.json`
- `.architec/architecture-contracts.json`
- `.architec/change-coupling.json`
- `.architec/runtime-paths.json`
- `.architec/style-profile.json`
- `.architec/architecture-review.json`

这些文件应成为 LLM 输入的主依据，而不是只把压缩后的摘要喂给模型。

## 8. 评分维度建议

现有分数偏重热点和 maintainability，后续建议扩展为以下维度：

- `topology_score`：仓库拓扑是否清晰、有组织
- `layering_score`：分层是否稳定、依赖是否单向
- `boundary_score`：公共接口和内部边界是否清晰
- `modularity_score`：模块大小、职责聚焦度是否合理
- `evolution_score`：新功能落点是否清晰，变更是否易扩散
- `style_consistency_score`：目录、命名、组织方式是否一致
- `operability_score`：配置流、运行链路、失败路径是否清楚
- `test_architecture_score`：测试是否支撑模块边界演进

最终总分不应只代表“代码整洁度”，而应代表“架构可持续性”。

## 9. 审查工作流建议

建议将完整架构审查拆成以下步骤：

1. 采集仓库证据  
读取拓扑、依赖、配置、入口、测试、变更耦合、静态指标。

2. 恢复架构模型  
识别层、组件、接口、边界、主流程和风格轮廓。

3. 执行规则审查  
检查结构违规、依赖违规、风格漂移和组织失衡。

4. 让 LLM 做架构师推理  
解释问题、评估风险、输出权衡和治理建议。

5. 生成治理结果  
输出评分、结论、建议、路线图、例外项和回归基线。

## 10. 报告形态建议

最终报告建议固定包含以下部分：

- 架构概览
- 当前架构形态判断
- 优点和稳定资产
- 主要结构性风险
- 架构发展风险
- 风格一致性问题
- 关键证据
- 分阶段治理建议
- 建议保持不动的部分
- 不确定项和人工复核点

这样报告才更像真人架构师的正式审查结果。

## 11. 分阶段落地路线

### Phase 1：补齐程序化结构审查

目标：

- 让工具能识别“项目铺开、缺少结构化管理”这类问题

建议内容：

- 新增 repo topology 分析
- 新增目录扁平度和文件归属稳定性指标
- 新增模块组织风格检查
- 在报告中显式展示结构化管理问题

### Phase 2：建立显式架构模型

目标：

- 让工具从“看文件”升级为“看架构”

建议内容：

- 输出 `architecture-model.json`
- 恢复层次、组件、接口和依赖规则
- 引入目标架构模板和项目级覆盖配置

### Phase 3：引入架构契约与漂移治理

目标：

- 让工具能够持续把控架构发展方向

建议内容：

- 支持层次依赖规则
- 支持公共接口/内部实现边界规则
- 支持例外、豁免、到期时间
- 支持架构漂移检测和趋势对比

### Phase 4：升级 LLM 为架构师推理层

目标：

- 让 LLM 负责判断、权衡和治理建议，而不是只负责摘要

建议内容：

- 为 LLM 提供结构化模型输入
- 固定架构审查问卷和输出 schema
- 输出置信度、争议点和人工复核点

## 12. 成功标准

当 `architec` 具备以下能力时，可以认为它开始接近“真正架构师”：

- 能指出“项目看起来散”的具体结构证据
- 能解释为什么散，以及会影响哪些未来变更
- 能判断问题属于目录组织、模块边界还是运行链路
- 能给出目标结构和迁移顺序
- 能在后续迭代中持续检测架构是否偏离目标
- 能统一项目的结构风格，而不只是统一代码格式

## 13. 结论

`architec` 的下一阶段重点，不应是继续增加摘要型输出，而应是：

- 把仓库恢复成显式架构模型
- 把架构发展和风格治理纳入程序化审查
- 让 LLM 基于证据完成架构师级判断

只有这样，`architec` 才不是“会写总结的分析工具”，而是“能持续把控项目架构发展的架构师代理”。
