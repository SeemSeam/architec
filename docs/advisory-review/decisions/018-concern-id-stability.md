# Concern ID Stability

Date: 2026-05-13

## Context

`code-review` concerns are consumed by humans, agents, and `fix-advice`. Early generated ids used source-local positions such as the first cleanup concern or first duplicate concern. That made ids easy to read, but a harmless ordering change could move the same file concern to a different id.

`fix-advice` accepts `--concern-id`, so the id should be stable enough to reference a concern fact from a saved review without encoding presentation rank.

## Decision

Generated `code-review` concern ids use deterministic ids based on concern facts rather than top-N or candidate order.

The v1 format is:

```text
code-review:<kind>:<12-char-hash>
```

The hash input includes the concern kind, source mapper, primary location, and factual evidence. For duplication concerns it also includes the duplicate location, reference location, and normalized fingerprint.

The id is intended to be stable across repeated runs when the same concern facts are produced. It is not guaranteed to survive file renames, symbol moves, evidence changes, or future mapper upgrades.

Existing saved review JSON with older ids remains valid input to `fix-advice`; consumers should not assume every historical id uses the hash format.

## Non-Goals

This does not:

- create a global registry of concern identities;
- guarantee ids across code moves or renamed files;
- make `concern_id` a severity or sorting signal;
- change the top-N ranking logic;
- change the JSON schema beyond id values generated by current code-review.

## Consequences

- `concern_id` no longer expresses display position.
- `fix-advice --concern-id` is safer for saved review files because repeated reviews of the same facts tend to keep the same id.
- Tests can reorder input candidates without changing ids for the same file concern.
- Future mappers should include stable factual inputs, not list indexes, when generating concern ids.
