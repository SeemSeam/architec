# 外部信号取舍

`architec` 可以读取历史、测试、依赖和运行时数据作为补充证据，但不重建专用工具能力。

## 历史信号

历史信号优先做：

- 每次 review 追加摘要到 `.architec/review-events.jsonl`。
- `status --trend` 展示维度趋势、反复恶化模块、显著变化来源。
- 增量审查可以提示“这是连续第 N 次让某维度下降”。
- 连续性提示只做事实陈述，不附加 `should stop`、`must address` 等导向语。

优先原因：

- 实现成本低。
- 价值会随使用时间累积。
- coding agent 通常缺少项目级长期记忆。

## 测试信号

测试信号轻度引入：

- 只读取已有 coverage 或测试映射报告。
- 关注高复杂度低测试、测试是否随代码同步变化、测试是否保护 public API。
- 不构建自己的测试执行框架。

下一阶段 test/churn risk fusion 应继续遵守这个边界：读取用户已有报告，把它们作为 code-review concern 的风险上下文，而不是由 `architec` 执行测试或生成覆盖率。

## 依赖信号

依赖信号只做架构视角：

- 新增依赖是否只在一处使用。
- 依赖是否扩大组件边界。
- 依赖是否绕过项目已有实现。
- 不做 CVE、license 或供应链扫描。

## 运行时信号

运行时信号暂缓：

- 不先做 APM、profiler 或线上错误采集。
- 后续只考虑读取用户已有报告，例如 production coverage 或 dead-code 报告。

## 更长期方向

- 历史相似方案比对：基于长期 review 事件流，提示当前方案是否类似过去曾发生偏离的方案。
- 运行时死代码或错误热点导入：只读取用户已有报告，不自建采集器。
- 更细粒度的深层耦合检测：隐式字段协议、共享状态和稳定性倒置的组合分析。
