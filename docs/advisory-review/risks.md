# 风险

## 主要风险

### 建议被误用为门禁

风险：

- 用户或 CI 把 `high-concern` 当作 `fail`。
- 一旦误伤，工具会被绕过。

缓解：

- 不输出 pass/fail 字段。
- 不提供 gate 命令。
- 文档明确 `architec` 不替用户做决定。

### 建议太多导致信号疲劳

风险：

- 每次 review 输出过多，用户或 agent 学会忽略。

缓解：

- 增量模式只谈本次变化。
- 默认突出 top N，建议 N 不超过 5。
- 支持按关注级别过滤。

### plan MD 解析不稳定

风险：

- 方案文档风格不一致，导致审查建议不可靠。

缓解：

- 提供推荐模板。
- 报告 `understood_plan`，让用户看到解析结果。
- 解析失败时降级为部分审查，不做过度推断。

### 与 coding agent 能力重叠

风险：

- coding agent 也能做局部代码审查，用户认为 `architec` 多余。

缓解：

- `architec` 专注项目级结构事实：组件边界、历史趋势、热点、cleanup 信号、方案一致性。
- 不做单函数优化建议。

### 旧命令迁移成本

风险：

- 用户已有脚本依赖旧命令。

缓解：

- breaking release 提供清晰迁移表。
- 可短期保留兼容警告，但文档只讲新命令。
